# v0.8.5 Trae Skills Consistency Audit

This document audits the Trae Skills against the current Factory design
principles.

Scope:

- Audit only
- No Skill edits
- No code edits
- No README edits
- No test edits

---

## Audit Baseline

Current Factory design principles:

- AI IDEs such as Trae, Codex, and ZCode are responsible for content generation.
- The Factory provides generation specifications, quality standards, JSON rules,
  validation, ZIP export, and local workflow structure.
- The Factory should not become an LLM API integration tool.
- The Factory should not require a runtime Prompt Layer implementation.
- The Factory should not adopt a content-engine approach.
- The Factory should not accumulate fixed educational content.
- Quality should be checked through `quality-report.json` and generation
  specifications, not hidden hardcoded content.

Primary reference documents:

- `docs/v0.8.4-generation-specification.md`
- `docs/v0.8.3-prompt-optimization-analysis.md`
- `docs/v0.8.2-automatic-regeneration-design.md`
- `README.md`
- `SOKQA_PACK_FACTORY_SPEC.md`

## Overall Result

The Skills are broadly aligned with the local-first Factory direction:

- They do not require Vercel, CDN, QR, manifest URL, or browser UI.
- They consistently describe local output under `output/<pack-id>/`.
- They use config-driven generation.
- They preserve multiple document/quiz support.
- They now consistently use `tts.choiceTexts`.
- They do not explicitly introduce content-engine architecture.
- They do not explicitly request fixed educational datasets.

However, several Skills still reflect older v0.5-era wording and do not fully
match the v0.8.2-v0.8.4 design:

- `llm-agents-orchestrator-sokqa` can be read as an LLM workflow implemented by
  the Factory, rather than an AI IDE orchestration guide.
- Retry count guidance says maximum 3 attempts, while v0.8.2 recommends default
  1 and hard max 2.
- Skills do not yet mention `quality-report.json`, quality score, or the current
  quality metrics.
- Profile fields are mostly absent from Skills.
- The improved v0.8.4 content-quality rules are not reflected in Skills.

No changes were made in this audit.

## Generation Specification Consistency

### Document Specification

Aligned:

- Multiple documents are controlled by `documentCount`.
- `doc_NN.json` naming is described.
- `language` is config-driven.
- Reference modes are listed in `sokqa-pack-factory-spec`.

Gaps:

- Skills do not include v0.8.4 document quality requirements:
  - learning order
  - distinct document roles
  - duplicate structure prevention
  - domain-specific examples
  - technical term explanation
  - target-user and difficulty effects

Risk:

- AI IDEs using only Skills may still generate structurally valid but shallow
  scaffold content.

### Quiz Specification

Aligned:

- `quizCount` controls quiz file count.
- `questionsPerQuiz` controls question count.
- `choices` must be exactly 4.
- `answerIndex` must be `0..3`.
- `tts.choiceTexts` is correctly described.

Gaps:

- Skills do not include v0.8.4 quiz quality requirements:
  - unique question viewpoints
  - theme-specific plausible distractors
  - answer position distribution
  - explanations covering correct and wrong choices
  - difficulty-specific reasoning depth
  - duplicate prevention across quiz files

Risk:

- AI IDEs may produce valid JSON but low-quality repeated quizzes.

### Metadata Specification

Aligned:

- `metadata.json` is required.
- `metadata.documents` and `metadata.quizzes` must match output files.
- `metadata.references` is described for multiple references.

Gaps:

- Skills do not mention `metadata.profile`.
- Skills do not list v0.6-v0.7 profile fields.
- Skills do not mention topic-specific tags or learner-facing metadata quality.

Risk:

- AI-generated packs may omit profile information or fail to use profile as a
  generation contract.

### TTS Specification

Aligned:

- `tts` fields are preserved.
- No audio files are generated.
- TTS structure is validated, not audio quality.
- `tts.choiceTexts` is the only quiz choice TTS field.
- Choice numbers are prohibited in `tts.choiceTexts`.

Gaps:

- Skills do not include v0.8.4 TTS quality requirements:
  - abbreviation reading
  - number reading
  - symbol reading
  - inline language switching
  - punctuation naturalness
  - avoiding label-only narration

Risk:

- TTS may be structurally valid but poor for Japanese or multilingual packs.

### Validation Specification

Aligned:

- JSON parse, type checks, required fields, counts, metadata consistency, ZIP
  consistency, and `exact_text_document` quiz suppression are represented.

Gaps:

- Skills do not mention `quality-report.json`.
- Skills do not mention current quality metrics:
  - `score`
  - `duplicateQuestionRate`
  - `duplicateDocumentRate`
  - `genericDistractors`
  - `genericExplanations`
  - `warnings`
- Skills do not include v0.8.2 retry target selection.

Risk:

- Users may treat structural validation as sufficient quality validation.

## Inconsistency List

### Critical

No Critical issues found.

Reasoning:

- No Skill currently instructs users to edit `lib`, add a content-engine, add
  fixed educational datasets, call an LLM API from Factory, or use the old
  `tts.choicesText` field.
- The existing issues can mislead future work, but they do not by themselves
  mandate a broken implementation path.

### Major

| ID | Area | Finding | Risk |
|---|---|---|---|
| M-1 | LLM ownership | `llm-agents-orchestrator-sokqa` uses "LLM workflow" and internal function names like `analyzeReference` and `buildPack`, which can be read as Factory runtime design rather than AI IDE workflow guidance. | May encourage adding LLM orchestration or prompt-layer code into Factory. |
| M-2 | Retry policy | `llm-agents-orchestrator-sokqa` says maximum 3 retries, while v0.8.2 recommends default 1 and hard max 2. | May create inconsistent regeneration behavior and higher loop/cost risk. |
| M-3 | Quality validation | Skills do not mention `quality-report.json`, `score`, duplicate rates, generic distractors, or generic explanations. | AI IDE users may stop at structural validation and miss content quality failures. |
| M-4 | Profile support | Skills do not mention `metadata.profile` or v0.6-v0.7 profile fields. | Generated output may ignore target user, difficulty, quiz style, and quality controls. |
| M-5 | v0.8.4 generation quality | Skills do not carry the improved document/quiz/TTS quality rules. | Agents using only Skills may generate valid but shallow packs. |

### Minor

| ID | Area | Finding | Risk |
|---|---|---|---|
| m-1 | Version wording | Skills still say spec version v0.5. | Confusing beside v0.8.x docs. |
| m-2 | Legacy naming | `nextjs-sokqa-pack-api` remains as a legacy Skill name. | Acceptable but visually inconsistent with local-first direction. |

## Priority

### Priority A

Recommended if Skills are updated next:

1. Reword `llm-agents-orchestrator-sokqa` so it clearly describes AI IDE
   orchestration, not Factory-embedded LLM execution.
2. Align retry guidance with v0.8.2:
   - default 1
   - hard max 2
   - no infinite retry
   - preserve final output and record failure
3. Add `quality-report.json` and quality metrics to validator/orchestrator Skills.
4. Add `metadata.profile` and profile-driven generation requirements.

### Priority B

Recommended after Priority A:

1. Bring v0.8.4 document quality rules into `sokqa-pack-factory-spec`.
2. Bring v0.8.4 quiz quality rules into `sokqa-pack-factory-spec` and
   `llm-agents-orchestrator-sokqa`.
3. Bring v0.8.4 TTS quality rules into `sokqa-json-shape-tts-validator`.
4. Expand reference-mode fidelity wording:
   - `source_only`: no outside facts
   - `exact_text_document`: no summary, no paraphrase, no quiz

### Priority C

Expression cleanup:

1. Update Skill version labels from v0.5 to a current or neutral wording.
2. Add a note explaining that `nextjs-sokqa-pack-api` is a legacy name only.
3. Clarify that `quality-report.json` is outside `learning-pack.zip`.
4. Repeat `schemaVersion: 1` consistently in Skill shape sections.

## Recommended Fix Content

No fixes were applied in this audit. If a follow-up update is requested, the
recommended direction is:

- Treat `.trae/skills` as AI IDE task guidance, not runtime architecture.
- Refer to `docs/v0.8.4-generation-specification.md` as the source of content
  generation quality.
- Refer to `docs/v0.8.2-automatic-regeneration-design.md` for retry limits and
  retry target selection.
- Keep the Factory local-first, config-driven, validation-driven, and ZIP-export
  focused.
- Avoid wording that suggests Factory should embed LLM API calls, prompt-layer
  runtime, content-engine modules, or fixed educational datasets.

---

## Audit Conclusion

The Skills are directionally consistent with the Factory's local-first,
config-driven workflow, and no Critical mismatch was found. The main consistency
problem is that the Skills lag behind v0.8.2-v0.8.4 design documents. They still
cover structure well, but they do not yet fully express the current quality,
profile, regeneration, and AI IDE ownership model.

The next Skill update should prioritize removing ambiguous LLM-runtime wording,
aligning retry limits, and importing the quality-report/Profile/generation-spec
requirements into the Skill documents.
