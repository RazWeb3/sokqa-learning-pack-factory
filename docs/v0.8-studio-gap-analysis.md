## 3. TTS Comparison

### 3.1 Multilingual / Code-Switching (CRITICAL GAP)

**Studio handles mixed Japanese + English natively** with inline language tags:

```
挨拶は時間帯によって...朝一番に誰かと会ったときは、誰もが知っている [en-US]Good morning[ja-JP] から始めましょう。
```

```
[en-US]I am sorry to hear that. [ja-JP]は、顧客がトラブルや不便に直面している...
```

The English phrase is read in English (`[en-US]`), then the explanation continues in Japanese (`[ja-JP]`). This is essential for language-learning packs.

**Factory** prefixes the **entire** TTS string with `[ja-JP]` and never code-switches. For an English conversation pack, the actual English phrase would be read with Japanese pronunciation rules — unusable.

### 3.2 Abbreviation / Reading Expansion

**Studio expands abbreviations to their spoken reading** in TTS:

| Original | Studio TTS |
|----------|------------|
| `IT` | `アイティー` |
| `ID` | `アイディー` |
| `URL` | `ユーアールエル` |
| `ECサイト` | `イーシーサイト` |
| `OS` | `オーエス` |
| `SSL/TLS` | `エスエスエル ティーエルエス` |
| `Wi-Fi` | `ワイファイ` |
| `RとL` | `アールとエル` |
| `VやB` | `ヴィーやビー` |
| `脆弱性` | `ぜいじゃくせい` |
| `漏洩` | `ろうえい` |
| `遵守` | `じゅんしゅ` |

**Factory** passes the raw text through `toTts()` unchanged — the TTS engine would receive `SSL/TLS`, `URL`, `Wi-Fi` as-is, which Japanese TTS reads poorly.

### 3.3 TTS Selectivity

**Studio does not emit TTS for every document item.** Only 12 of 20 items in the IT security doc have `tts.text`. Items without TTS are presumably visual-only (diagrams, lists, headers).

**Factory** emits TTS for every item mechanically. Combined with the double-punctuation bug (`、,`), this produces a wall of low-quality TTS.

### 3.4 TTS Punctuation

| Studio | Factory |
|--------|---------|
| Clean sentence, ends with `。` or natural punctuation | Ends with `、,` (Japanese comma + English comma) — double punctuation |

### 3.5 TTS Field Coverage

| Field | Studio | Factory |
|-------|--------|---------|
| `tts.text` (documents) | Selective, high-quality, multilingual | All items, single-language, raw |
| `tts.questionText` (quiz) | Not shown in samples (likely omitted when question is simple) | Always present, mechanical |
| `tts.choiceTexts` (quiz) | Not shown in samples | Always present, all identical pattern |
| `tts.answerText` (quiz) | Not shown | Always present |
| `tts.explanationText` (quiz) | Selective, multilingual with `[en-US]`/`[ja-JP]` | Always present, single-language |

Studio appears to **emit TTS only where it adds value** (explanations with English phrases, document narration). Factory emits TTS everywhere uniformly.

## 5. Cross-Cutting Findings

### 5.1 The Core Gap: Content vs. Scaffolding

The fundamental difference is that **Studio generates content; Factory generates scaffolding.**

- Studio writes 20 distinct paragraphs explaining real security concepts.
- Factory writes 9 template slots filled with the theme string.

This single difference accounts for the vast majority of the quality gap. All other features (TTS quality, distractor quality, tag richness) are secondary to this.

### 5.2 The Factory's Strength Is Also Its Weakness

Factory's `profile` system (14 generation controls) is a strength because it makes generation configurable. But it is also a weakness because the controls adjust **template wording** (e.g., `difficulty: hard` adds the word "applied" to the correct answer) rather than **content substance**. The profile system has no content to act upon.

### 5.3 Reference Mode Is the Bridge

Factory's reference modes (`source_only`, `source_plus`, `exact_text_document`) are the only path to Studio-quality output today. When a real reference document is provided, Factory can produce content-rich packs. The gap is specifically in **no-reference (`mode: none`) generation**, where Factory falls back to empty templates.

## 6. v0.8 Implementation Priority

### CRITICAL (blocks usefulness)

| Priority | Item | Rationale |
|----------|------|-----------|
| C1 | **Content generation engine for no-reference mode** (F1) | Without this, `mode: none` produces unusable packs. Every other improvement is cosmetic if the content is empty. |
| C2 | **Domain-specific distractor generation** (F11) | Quizzes are only useful if distractors require domain knowledge to reject. Current meta-patterns are trivially identifiable as wrong. |
| C3 | **Per-question variety** (F2) | A 5-question quiz with 1 unique question has no learning value. Need distinct questions per slot and across quizzes. |

### HIGH (quality differentiators)

| Priority | Item | Rationale |
|----------|------|-----------|
| H1 | **Inline multilingual TTS** `[en-US]...[ja-JP]` (F3) | Language-learning packs are unusable without code-switching. Directly blocks the English conversation use case. |
| H2 | **Abbreviation-to-reading expansion** (F4) | Japanese TTS reads `URL`, `SSL`, `IT`, `Wi-Fi` poorly. A mapping table fixes this for IT/technical themes. |
| H3 | **Answer index shuffling** (F10) | "Always index 0" makes quizzes trivially guessable. Trivial to fix, high fairness impact. |
| H4 | **TTS double-punctuation fix** (`、,`) | Quality bug already noted in preflight report. |

### MEDIUM (polish and parity)

| Priority | Item | Rationale |
|----------|------|-----------|
| M1 | **Selective TTS emission** (F5) | Reduce noise; emit TTS only where it adds value. |
| M2 | **Topic-specific `globalTags`** (F6) | Derive 2-3 tags from theme/title for metadata richness. |
| M3 | **Chapter-anchored titles** (F9) | `"セキュリティ基礎 1. ..."` style improves discoverability. |
| M4 | **`author` + `assetBaseUrl` schema fields** (F7, F8) | Trivial schema parity with Studio. |
| M5 | **Learning-arc document structure** (F12) | Replace fixed template with topic-aware pedagogical sections. Depends on C1. |

### LOW (nice-to-have)

| Priority | Item | Rationale |
|----------|------|-----------|
| L1 | **Preset profiles** | Reduce config verbosity (already in preflight report). |

## 7. Summary

### Studio's Strengths

1. **Generates real educational content** — paragraphs that teach concepts, not templates that mention the theme.
2. **Quiz questions are distinct and domain-specific** — 20 unique questions covering topic breadth, with plausible domain distractors.
3. **TTS is multilingual and abbreviation-aware** — code-switches `[en-US]`/`[ja-JP]` and expands `IT→アイティー`.
4. **Metadata is rich** — topic tags, author, assetBaseUrl, chapter-anchored titles.
5. **Produces variety on the same topic** — two CS English quiz variants differ throughout.

### Factory's Weaknesses

1. **No-reference mode produces empty scaffolding** — theme string inserted into templates, no real content.
2. **Quiz questions are duplicates** — 80%+ repetition within and across quizzes.
3. **Distractors are meta-patterns** — obviously wrong, require no domain knowledge.
4. **TTS is single-language and raw** — no code-switching, no abbreviation expansion, double punctuation `、,`.
5. **Answer always at index 0** — trivially guessable.
6. **Metadata is thin** — single generator tag, no topic tags, no author/assetBaseUrl.

### The Path to Parity

The gap is large but **structurally clear**. The Factory's config/profile/validation/ZIP infrastructure is solid — what it lacks is a **content generation engine**. Once C1 (content generation for no-reference mode) is in place, the HIGH items (multilingual TTS, abbreviation expansion, answer shuffling) become incremental polish on real content rather than polish on empty templates.

**Recommended v0.8 focus: C1 + C2 + C3 + H1 + H2 + H3 + H4.** Defer M-tier items to v0.9 once the content engine proves out.
