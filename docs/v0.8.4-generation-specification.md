# v0.8.4 Generation Specification

This document defines an improved Generation Specification for AI IDEs such as
Trae, Codex, ZCode, and similar agent workflows.

The Factory itself is not an LLM API integration tool. Its role is to provide:

- Generation rules
- Quality standards
- JSON schema expectations
- Validation expectations
- Export-compatible learning-pack structure

Scope for v0.8.4:

- Specification and analysis only
- No implementation
- No code changes
- No README changes
- No test changes

---

## Positioning

AI IDEs generate the learning content. The Factory validates, packages, and
exports SokQA-compatible files. Therefore, the generation specification must be
clear enough that different AI agents can target the same quality level without
depending on hidden prompt behavior.

The specification below should be usable as a direct instruction document for an
AI generation agent.

## Current Specification Inventory

### Document

Current required conditions:

- Output `doc_NN.json` files according to `documentCount`.
- Each document file has `type: "document"`.
- Each document file has `schemaVersion: 1`.
- `language` must match config and metadata.
- `documents[]` must contain at least one item.
- Each item has `id`, `text`, and `tts.text`.
- Item ids use `doc-N` shape.

Current recommended conditions:

- Use `theme`, `title`, and `description`.
- Respect reference modes:
  - `none`
  - `source_only`
  - `source_plus`
  - `exact_text_document`
- Preserve exact source text in `exact_text_document`.
- In `source_only`, avoid adding non-reference facts.
- In `source_plus`, center the reference and add learner-friendly support.

Missing or weak conditions:

- Learning order is not specified strongly enough.
- Chapter/document roles are not required.
- Duplicate document structure is not prohibited.
- Examples are not required to be domain-specific.
- Technical terms do not require explanation.
- Difficulty and target user are not mapped to concrete content behavior.
- Summary quality is not defined.

### Quiz

Current required conditions:

- Output `quiz_NN.json` files according to `quizCount`.
- Each quiz file has `type: "quiz"`.
- Each quiz file has `schemaVersion: 1`.
- `language` must match config and metadata.
- Each quiz has `questions[]`.
- Each question has `id`, `question`, `choices`, `answerIndex`,
  `explanation`, and `tts`.
- `choices` must contain exactly 4 items.
- `answerIndex` must be `0..3`.
- Each quiz must contain `questionsPerQuiz` questions.
- `exact_text_document` generates no quiz.

Current recommended conditions:

- `quizStyle` may be `concept-check`, `application`, or `case-study`.
- `distractorSource` may be `fixed`, `theme`, or `reference`.
- `explanationDepth` may be `short`, `standard`, or `detailed`.
- `tts.choiceTexts` mirrors `choices`.
- `tts.choiceTexts` must not include option numbers.

Missing or weak conditions:

- Question viewpoints are not required to be unique.
- Distractor plausibility is not defined.
- Generic wrong choices are not explicitly prohibited.
- Explanations are not required to cover both correct and wrong answers.
- Answer positions are not required to be distributed.
- Difficulty is not mapped to question reasoning depth.
- Cross-quiz duplication is not prohibited.

### Metadata

Current required conditions:

- `metadata.json` is required.
- Required fields include `id`, `title`, `description`, `createdAt`,
  `generator`, `version`, `language`, `source`, `profile`, `references`,
  `documents`, and `quizzes`.
- `metadata.documents` must match generated document file names.
- `metadata.quizzes` must match generated quiz file names.
- `metadata.references` must match reference input file names.
- `source.hasReference`, `source.mode`, and `source.path` must be consistent
  with config.

Current recommended conditions:

- Record resolved profile in `metadata.profile`.
- Keep metadata as the manifest-like local index for output.
- Do not require CDN, QR, Vercel, manifest URL, or hosted asset URLs.

Missing or weak conditions:

- Topic-specific tags are not required.
- Chapter-specific document and quiz titles are not required.
- Learner-facing description quality is not defined.
- Metadata does not define intended audience beyond `profile`.

### TTS

Current required conditions:

- Keep `tts` fields.
- Do not generate audio files.
- Validate TTS structure and presence, not audio quality.
- `tts.choiceTexts` must be a 4-item array.
- `tts.choiceTexts` must match the order of `choices`.
- `tts.choiceTexts` must not include choice numbers.
- Quiz TTS choice fields must use `tts.choiceTexts` only.

Current recommended conditions:

- Use language-aware text.
- Use `choiceTexts` as choice-body-only narration.

Missing or weak conditions:

- Abbreviation reading is not specified.
- Number reading is not specified.
- Symbol reading is not specified.
- Inline language switching is not specified.
- Unnecessary narration is not defined.
- Punctuation rules are not defined.
- Studio-like multilingual TTS behavior is not captured.

### Validation

Current required conditions:

- JSON parse succeeds.
- Required fields exist.
- Types are valid.
- Document, quiz, and metadata shape is valid.
- Counts match config:
  - `documentCount`
  - `quizCount`
  - `questionsPerQuiz`
- Metadata file lists match generated files.
- ZIP file list matches metadata.
- `answerIndex` is in range.
- `choices.length === 4`.
- `tts.choiceTexts` contains no choice numbering.
- `exact_text_document` has no quizzes.

Current recommended conditions:

- Run programmatic validation before export is considered successful.
- Write `quality-report.json` after generation.
- Use quality metrics to detect duplicate and generic output.

Missing or weak conditions:

- Content-level quality pass/fail rules are not part of the base generation
  specification.
- AI agents do not yet have a shared preflight checklist.
- Regeneration triggers are split across docs and skills, not consolidated into
  one AI-facing spec.

## Document Specification

### Required Structure

An AI generator must produce exactly `documentCount` document files unless
`documentCount` is invalid before generation.

Each document file must contain:

```json
{
  "id": "pack-id_doc_01",
  "type": "document",
  "schemaVersion": 1,
  "title": "Document title",
  "description": "Document description",
  "language": "ja",
  "globalTags": ["tag"],
  "documents": [
    {
      "id": "doc-1",
      "text": "Display text",
      "tts": {
        "text": "Spoken text"
      }
    }
  ]
}
```

Required content rules:

- `language` must match config.
- `documents[]` must not be empty.
- Each item must teach or reinforce one clear learning point.
- Each document must have a distinct role when `documentCount > 1`.
- Do not create multiple documents with the same section order and only changed
  titles.

### Learning Order

Documents should follow a learner-friendly order:

1. Context: what the learner is studying and why it matters.
2. Basics: key terms, facts, or concepts.
3. Structure: how the concepts relate to each other.
4. Application: what the learner should do with the knowledge.
5. Examples: concrete theme-specific situations.
6. Common mistakes: misconceptions or traps.
7. Summary: concise review of what should be remembered.

For multiple documents, assign roles before writing:

| Document | Typical role |
|---|---|
| `doc_01` | Basics and vocabulary |
| `doc_02` | Applied understanding or workflow |
| `doc_03` | Practical cases, mistakes, and review |

The exact roles may vary by theme, but they must not duplicate each other.

### Chapter Structure

Each document should have a coherent chapter-like title:

- Good: `情報セキュリティ 1. 三要素と身近なリスク`
- Weak: `ITパスポート入門 (01)`

Each document item should be self-contained enough to be useful in isolation.
Avoid label-only items such as:

```text
Theme: ITパスポート
Goal: ITパスポート入門
Key Points: ITパスポートの基礎
```

### Duplicate Prevention

Do not repeat:

- Same heading sequence across documents
- Same example pattern across documents
- Same summary wording across documents
- Same paragraph with only the theme replaced
- Same document role under different titles

When the same concept must appear more than once, change the purpose:

- First mention: definition
- Later mention: application
- Later mention: common mistake or review

### Examples

Examples must be specific to the theme, target user, and output mode.

Good examples:

- For IT security: phishing email, password reuse, MFA, OS update, URL spoofing.
- For prefectures: capitals, regions, local specialties, neighboring areas.
- For customer support: complaint handling, delivery delay, refund policy.

Weak examples:

- `Apply the theme in a practical scenario.`
- `Use the topic correctly.`
- Customer-service examples in an unrelated topic.

If `practicalExamples` is `true`, include real-life examples or realistic
mini-scenarios. If `false`, examples may be fewer, but any included example
still must be theme-specific.

### Technical Terms

When using specialized terms:

- Define the term the first time it appears.
- Use the configured language.
- Use a simpler explanation for `targetUser: beginner`.
- Use more precise terminology for `targetUser: student`, `employee`, or
  `manager` when appropriate.
- For difficult Japanese readings or abbreviations, reflect pronunciation in
  TTS, not necessarily in display text.

### Summary

Every document should end with a useful summary or review item unless the
reference mode prohibits additions.

A good summary:

- Names the main concepts.
- States what the learner should remember.
- Avoids generic praise-only wording.

Weak:

```text
Nice work! You covered the theme.
```

Better:

```text
情報セキュリティでは、機密性・完全性・可用性を守るために、認証、更新、暗号化、バックアップを組み合わせて考えることが重要です。
```

### Reference Modes

| Mode | Document rule |
|---|---|
| `none` | Generate from theme/title/description. Use general knowledge carefully and keep output educational. |
| `source_only` | Use only reference facts. Do not add outside facts. Reorganize only when it does not change meaning. |
| `source_plus` | Center reference facts and add learner-friendly explanation, examples, or context. |
| `exact_text_document` | Preserve original text. Do not summarize, paraphrase, or add quiz. |

## Quiz Specification

### Required Structure

An AI generator must produce exactly `quizCount` quiz files except in
`exact_text_document`, where quiz count must be treated as zero.

Each quiz question must contain:

```json
{
  "id": "q-1",
  "question": "Question text",
  "choices": ["A", "B", "C", "D"],
  "answerIndex": 2,
  "explanation": "Explanation text",
  "tts": {
    "questionText": "Question spoken text",
    "choiceTexts": ["A spoken", "B spoken", "C spoken", "D spoken"],
    "answerText": "Answer spoken text",
    "explanationText": "Explanation spoken text"
  }
}
```

Required content rules:

- Each quiz has exactly `questionsPerQuiz` questions.
- Each question has exactly 4 choices.
- Exactly one choice is correct.
- `answerIndex` points to the correct choice.
- `answerIndex` must not be always the same value.
- Questions in the same quiz must test different viewpoints.
- Questions across quiz files should not repeat the same stem, choices, and
  explanation.

### Question Viewpoints

Before writing questions, assign each question a unique viewpoint:

- Definition
- Relationship
- Process order
- Cause and effect
- Example identification
- Correct action
- Common misconception
- Case judgment
- Comparison between similar concepts
- Interpretation of reference material

Do not create a quiz where every question asks:

```text
Which choice best matches the theme?
```

### Distractors

Wrong choices must be plausible but clearly wrong.

Required distractor rules:

- Use the same domain category as the correct answer.
- Avoid obvious meta-wrong options.
- Avoid options that are wrong only because they sound careless.
- Avoid "all of the above" unless explicitly intended and supported.
- Avoid making the correct answer much longer, more detailed, or more positive
  than all distractors.

Examples:

| Theme | Correct | Good distractors | Weak distractors |
|---|---|---|---|
| 47 prefectures | `47` | `43`, `45`, `50` | `Ignore the question` |
| Security | `MFA` | `password reuse`, `plain HTTP`, `shared account` | `Use unclear wording` |
| English phrase | `Thank you for contacting us.` | `Thanks for calling me.`, `Good to see you.` | `Skip the basics` |

### Explanation

Every explanation must teach.

Minimum explanation rules:

- State why the correct answer is correct.
- State why at least one distractor is wrong.
- Use theme-specific terms.
- Avoid only saying "best matches the theme."

By `explanationDepth`:

| Value | Required behavior |
|---|---|
| `short` | One concise sentence explaining the correct answer. |
| `standard` | Explain the correct answer and at least one wrong option. |
| `detailed` | Explain the correct answer and why each wrong option fails. |

### Answer Position

For quizzes with 4 or more questions:

- Use at least 3 different answer positions.
- Avoid predictable patterns such as `0,1,2,3,0,1,2,3` when possible.
- Do not make `answerIndex` always `0`.

For smaller quizzes:

- Still avoid placing every answer in the same position.

### Difficulty

| Difficulty | Quiz behavior |
|---|---|
| `easy` | Direct recognition of basic facts and definitions. Distractors should be clearly related but easier to reject. |
| `normal` | One-step application, comparison, or interpretation. Distractors should reflect common confusion. |
| `hard` | Scenario-based reasoning, subtle distinctions, or multi-step judgment. Distractors should be plausible traps. |

### Quiz Style

| quizStyle | Required behavior |
|---|---|
| `concept-check` | Test terms, facts, definitions, relationships, and recognition. |
| `application` | Ask what the learner should do in a realistic situation. |
| `case-study` | Present a short case and ask for the best judgment or next action. |

## Metadata Specification

### Required Structure

`metadata.json` must include:

- `id`
- `title`
- `description`
- `createdAt`
- `generator`
- `version`
- `language`
- `source.hasReference`
- `source.mode`
- `source.path`
- `profile`
- `references`
- `documents`
- `quizzes`

The file lists must match actual generated output:

- `documents`: `["doc_01.json", ...]`
- `quizzes`: `["quiz_01.json", ...]`

For `exact_text_document`, `quizzes` must be empty.

### Recommended Metadata Quality

AI-generated metadata should improve discoverability:

- `title` should be learner-facing and concise.
- `description` should describe what the learner can do after completion.
- Document and quiz titles should reflect roles or chapters.
- `globalTags` inside document/quiz files should include topic-specific tags,
  not only generator tags.

Do not require:

- CDN URL
- QR code
- Vercel URL
- Hosted manifest URL
- Audio file paths

## TTS Specification

### Required Shape

Document items:

```json
"tts": {
  "text": "Spoken document text"
}
```

Quiz questions:

```json
"tts": {
  "questionText": "Spoken question text",
  "choiceTexts": ["Choice A", "Choice B", "Choice C", "Choice D"],
  "answerText": "Spoken answer text",
  "explanationText": "Spoken explanation text"
}
```

### choiceTexts Rule

`tts.choiceTexts` must:

- Be a 4-item array.
- Mirror `choices` in the same order.
- Contain only the spoken choice body.
- Not include `1番`, `2番`, `Option 1`, `Choice 1`, `No.1`, or similar.
- Use `tts.choiceTexts` as the only quiz TTS choice field.

### Spoken Text Rules

TTS text is not merely display text copied into another field. It should be
spoken text.

Required rules:

- Use natural pronunciation for the configured language.
- Avoid label-only narration such as `Theme`, `Goal`, or `Example 1`.
- Avoid double punctuation.
- Avoid unnatural comma suffixes.
- Keep sentence endings natural for the language.

### Abbreviations

For Japanese TTS, expand common abbreviations into readable Japanese phonetics
when they appear in spoken text.

Examples:

| Display | TTS guidance |
|---|---|
| `IT` | `アイティー` |
| `URL` | `ユーアールエル` |
| `ID` | `アイディー` |
| `OS` | `オーエス` |
| `Wi-Fi` | `ワイファイ` |
| `SSL/TLS` | `エスエスエル ティーエルエス` |

Do not over-expand common words when the configured language can read them
naturally.

### Numbers

Read numbers according to context:

- Counts: natural count reading.
- Years: year reading.
- Percentages: include percent reading.
- Ordered steps: do not add option numbering inside `choiceTexts`.
- Code-like identifiers: preserve or spell out only when needed for clarity.

### Symbols

Symbols should be spoken naturally:

- `/` in technical abbreviations may need expansion.
- `&` should become "and" or the target-language equivalent.
- Parentheses should usually be omitted or paraphrased in TTS.
- Long URLs should be avoided in TTS unless the lesson specifically teaches URLs.

### Multilingual and Inline Language Switching

Studio-quality TTS uses inline language tags when content switches language.

Example:

```text
[ja-JP]朝の挨拶では、[en-US]Good morning[ja-JP] を使います。
```

Rules:

- Use the configured language as the default.
- Add inline language tags for foreign phrases that should be pronounced in that
  language.
- For English-learning content in Japanese, English phrases should generally use
  `[en-US]`.
- Do not wrap the entire string in Japanese when the key phrase is English.

### Unnecessary Reading

Avoid TTS for content that is only structural if the schema or downstream app
allows omission in the future. Under the current schema, keep the field present
but write a natural spoken equivalent.

Weak:

```text
[ja-JP]Theme、
```

Better:

```text
[ja-JP]この章では、情報セキュリティの基本を学びます。
```

### Studio Difference

Studio examples show:

- Inline `[en-US]` / `[ja-JP]` switching.
- Abbreviation expansion.
- Selective TTS where narration adds value.
- Natural punctuation.

Factory specification should move toward those behaviors while keeping current
JSON compatibility.

## Validation Specification

### Required Validation

AI output must pass these checks before export:

- Valid JSON.
- Required fields exist.
- Types match schema.
- Document count matches `documentCount`.
- Quiz count matches `quizCount`.
- Each quiz has `questionsPerQuiz` questions.
- `exact_text_document` has no quizzes.
- Each quiz question has exactly 4 choices.
- `answerIndex` is `0..3`.
- `tts.choiceTexts` has exactly 4 items.
- `tts.choiceTexts` contains no option numbers.
- Metadata file lists match generated files.
- ZIP file list matches metadata.
- References recorded in metadata match reference inputs.

### Quality Validation

AI output should also satisfy quality checks:

- No high duplicate question rate.
- No high duplicate document rate.
- No generic distractors.
- No generic explanations.
- No repeated choice sets.
- No repeated document section structure.
- No always-zero answer index.
- No language mismatch between config and generated content.
- No unnatural TTS punctuation.

### Regeneration Required

Regenerate the affected part when:

- JSON is invalid.
- Required fields are missing.
- Counts do not match config.
- `answerIndex` is invalid.
- `choiceTexts` contains option numbers.
- The same question viewpoint repeats.
- Wrong choices are generic or obviously wrong.
- Explanations only restate the theme.
- Multiple documents have the same content or structure.
- TTS is unreadable, has wrong language tags, or contains double punctuation.

Regeneration target:

| Problem | Target |
|---|---|
| Document duplication | Document only |
| Quiz duplication | Quiz only |
| Generic distractors | Quiz only |
| Generic explanations | Quiz only |
| Metadata file mismatch | Metadata and export list |
| Cross-file incoherence | Full pack |

## AI Generation Guidelines

This section can be copied directly into an AI IDE generation task.

### Always Follow

- Generate SokQA-compatible `document`, `quiz`, and `metadata` JSON.
- Follow config counts exactly.
- Use config `language`.
- Use config `theme`, `title`, and `description`.
- Respect reference mode.
- Keep JSON valid.
- Keep `tts` fields.
- Use `tts.choiceTexts` as the only quiz TTS choice field.
- Do not include choice numbers in `tts.choiceTexts`.
- Make each document and quiz item educational, not merely structural.
- Make each quiz question test a distinct viewpoint.
- Make distractors plausible and theme-specific.
- Explain correct and wrong answers according to `explanationDepth`.
- Distribute `answerIndex`.

### Prohibited

- Do not output markdown instead of JSON when JSON is requested.
- Do not use a fixed 5-question assumption; use `questionsPerQuiz`.
- Do not use a fixed English language assumption; use config `language`.
- Do not generate `quiz` files for `exact_text_document`.
- Do not invent facts in `source_only`.
- Do not paraphrase in `exact_text_document`.
- Do not use generic distractors such as "skip the basics" unless the topic is
  explicitly about learning behavior.
- Do not make every answer index `0`.
- Do not repeat the same question with shuffled choices.
- Do not create documents that differ only by title.
- Do not include option numbers in `tts.choiceTexts`.
- Do not rely on CDN, QR, Vercel, or hosted URLs.

### Quality Standard

Generated output should meet this bar:

- A first-time learner can understand the flow without external explanation.
- Each document has a clear role.
- Each document item teaches a concrete point.
- Each quiz question tests a unique point.
- Distractors are believable.
- Explanations teach.
- TTS is readable and language-aware.
- Metadata accurately describes the generated files.

### Recommended Generation Steps

1. Read config and reference settings.
2. Create a brief content plan:
   - document roles
   - quiz viewpoints
   - key terms
   - examples
   - TTS pronunciation notes
3. Generate document JSON.
4. Generate quiz JSON.
5. Generate metadata JSON.
6. Self-check counts, schema, duplication, answer positions, and TTS.
7. Fix issues before export.

## Profile Reflection Status

### Current Required Mapping

| Profile field | Must influence |
|---|---|
| `targetUser` | Vocabulary, assumed prior knowledge, example context, explanation style. |
| `difficulty` | Depth, reasoning complexity, distractor closeness, case complexity. |
| `learningStyle` | Document shape, TTS density, review style, quiz preparation cues. |
| `outputMode` | Study/training/exam framing and question style. |
| `tone` | Wording style across documents and explanations. |
| `quizStyle` | Question format and scenario type. |
| `distractorSource` | Source of wrong answers. |
| `explanationDepth` | Amount of reasoning in explanations. |
| `practicalExamples` | Whether realistic examples are required. |
| `sentenceLength` | Display and TTS sentence length. |

### Detailed Profile Rules

| Field | Values | Generation rule |
|---|---|---|
| `targetUser` | `beginner` | Explain terms, avoid assumptions, use simple examples. |
| `targetUser` | `student` | Use study-oriented terminology and memory cues. |
| `targetUser` | `employee` | Use workplace examples and practical actions. |
| `targetUser` | `manager` | Emphasize decision impact, risk, process, and oversight. |
| `targetUser` | `general` | Use everyday wording and broadly understandable examples. |
| `difficulty` | `easy` | Basic facts, direct recognition, simple distractors. |
| `difficulty` | `normal` | One-step application and common confusions. |
| `difficulty` | `hard` | Case judgment, subtle distinctions, plausible traps. |
| `learningStyle` | `audio` | Short, spoken-friendly sentences and natural TTS. |
| `learningStyle` | `reading` | Richer paragraphs and connected explanation. |
| `learningStyle` | `quiz` | Facts organized for recall and review. |
| `learningStyle` | `balanced` | Mix explanation, examples, and review. |
| `outputMode` | `study` | Build understanding and retention. |
| `outputMode` | `training` | Emphasize workplace action and scenarios. |
| `outputMode` | `exam` | Include exam-like checks and precise distinctions. |
| `tone` | `friendly` | Plain, encouraging, but not shallow. |
| `tone` | `professional` | Direct, operational, and concise. |
| `tone` | `academic` | Formal, concept-oriented, and precise. |
| `quizStyle` | `concept-check` | Definitions, relationships, core facts. |
| `quizStyle` | `application` | Realistic use and next action. |
| `quizStyle` | `case-study` | Short case and best judgment. |
| `distractorSource` | `fixed` | Legacy-safe fallback only; avoid if quality matters. |
| `distractorSource` | `theme` | Generate wrong answers from theme-specific concepts. |
| `distractorSource` | `reference` | Draw plausible wrong choices from reference context without violating source constraints. |
| `explanationDepth` | `short` | One concise reason. |
| `explanationDepth` | `standard` | Correct reason plus one wrong-choice contrast. |
| `explanationDepth` | `detailed` | Correct reason plus all wrong-choice contrasts. |
| `practicalExamples` | `true` | Include realistic theme-specific examples. |
| `practicalExamples` | `false` | Examples optional, but no generic placeholders. |
| `sentenceLength` | `short` | Short display/TTS sentences; avoid fragment-only TTS. |
| `sentenceLength` | `medium` | Standard readable sentence length. |
| `sentenceLength` | `long` | Longer explanations allowed when useful. |

### Gaps to Address

- `targetUser` must affect content depth, not only labels.
- `difficulty` must affect reasoning, not only wording.
- `learningStyle: quiz` should change document structure toward review-ready
  facts.
- `distractorSource: theme` must produce domain wrong answers, not generic bad
  behaviors.
- `sentenceLength` should control generated text, not post-generation truncation.
- `tone` should remain consistent across all generated fields.

## v0.8.4 Decision Summary

- The Factory remains local-first and not an LLM API integration tool.
- AI IDEs are responsible for content generation.
- The Generation Specification must define both JSON shape and content quality.
- Document generation must require learning order, distinct roles, examples,
  summaries, term explanations, and profile-aware depth.
- Quiz generation must require unique viewpoints, plausible distractors,
  answer-index distribution, and useful explanations.
- TTS generation must handle abbreviations, numbers, symbols, multilingual
  content, inline language switching, unnecessary narration, and punctuation.
- Validation must include both structural checks and quality-triggered
  regeneration guidance.
