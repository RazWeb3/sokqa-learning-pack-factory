# v0.8.2 Automatic Regeneration Design

This document designs the future automatic regeneration workflow for low-quality
learning-pack output.

Scope for v0.8.2:

- Design only
- No implementation
- No CLI behavior change
- No README update
- No test update

---

## Background

v0.8 added `quality-report.json` and a numeric `score` so generated packs can be
reviewed after output. The next step is to decide when the generator should try
again, what part it should regenerate, how many times it may retry, and how the
retry result should be recorded.

The goal is not to chase a perfect score. The goal is to make low-quality output
recoverable without causing confusing behavior, unbounded LLM cost, or infinite
generation loops.

---

## Current quality-report specification

Current implementation source:

- `lib/quality-check.ts`
- `lib/learning-pack-zip.ts`
- `README.md`

`quality-report.json` is written next to `learning-pack.zip` in the output
directory. It is not included in the ZIP.

Current structure:

```json
{
  "score": 92,
  "duplicateQuestionRate": 5,
  "duplicateDocumentRate": 3,
  "genericDistractors": 1,
  "genericExplanations": 0,
  "warnings": [
    "duplicate-question: q-1 ~ q-2 (question=91%, explanation=88%, choices=true)"
  ]
}
```

### score

The score starts at `100` and subtracts penalties. The final value is rounded and
clamped to `0..100`.

Current formula:

```text
score =
  100
  - duplicateQuestionRate * 1
  - duplicateDocumentRate * 1
  - genericDistractors * 5
  - genericExplanations * 5
```

### duplicateQuestionRate

`duplicateQuestionRate` is the percent of quiz questions flagged as duplicates.
A pair of questions is considered duplicate-like when any of the following is
true:

- Question text similarity is `>= 0.8`
- Explanation similarity is `>= 0.8`
- The normalized choice set is identical

Warning prefix:

```text
duplicate-question:
```

### duplicateDocumentRate

`duplicateDocumentRate` is the percent of documents flagged as duplicate-like.
A pair of documents is considered duplicate-like when any of the following is
true:

- Full document text similarity is `>= 0.8`
- Titles differ but text similarity is `>= 0.95`
- Extracted section structure is identical

Warning prefix:

```text
duplicate-document:
```

### genericDistractors

`genericDistractors` counts questions that contain template-like wrong choices.
Current generic patterns are:

- `skip the basics`
- `apply without checking`
- `use unclear wording`

Warning prefix:

```text
generic-distractor:
```

### genericExplanations

`genericExplanations` counts questions with template-like explanations or
explanations that do not contain theme keywords.

An explanation is treated as generic when:

- It is empty or effectively template-only
- It does not include any extracted theme keyword

Current generic explanation words:

- `best`
- `matches`
- `bestmatches`
- `reason`
- `why`

Warning prefix:

```text
generic-explanation:
```

### warnings

`warnings` is a flat array of human-readable strings. It currently records the
specific duplicate pairs or generic-content findings. The strings are useful for
debugging but should not become the only machine-readable retry contract.

## Pass line

Recommended future status mapping:

| Score range | Status | Meaning |
|---|---|---|
| `score >= 80` | `pass` | Good enough. No automatic retry by default. |
| `60 <= score < 80` | `warning` | Usable but quality concerns exist. Retry only when explicitly enabled. |
| `score < 60` | `retry_candidate` | Low quality. Retry when automatic retry is enabled. |

Recommended default:

- Generate once
- Evaluate quality
- Retry only if `--quality-retry` is enabled and the result is `retry_candidate`

Rationale:

- A score below 60 usually means repeated questions/documents or multiple generic
  quiz fields, which is severe enough to justify retry cost.
- Warning-level output may still be usable, especially for drafts or local
  iteration. Retrying every warning could surprise users with longer runs and
  higher LLM usage.

## Regeneration targets

Automatic regeneration should choose the smallest target that can plausibly fix
the quality issue.

### document-only regeneration

Use when document quality is the main issue and quiz quality is acceptable.

Primary triggers:

- `duplicateDocumentRate` is high
- Warnings are dominated by `duplicate-document:`

Recommended threshold:

```text
duplicateDocumentRate >= 50
```

Output behavior:

- Regenerate document files
- Preserve quiz files when possible
- Rebuild metadata and ZIP after document regeneration
- Re-run quality report on the merged result

Prompt direction:

- Separate document roles
- Avoid reusing the same section headings and body structure
- Allocate the theme into distinct angles, such as basics, applied use, and
  practical examples

### quiz-only regeneration

Use when quiz quality is the main issue and documents are acceptable.

Primary triggers:

- `duplicateQuestionRate` is high
- `genericDistractors` is greater than zero
- `genericExplanations` is greater than zero
- Warnings are dominated by `duplicate-question:`, `generic-distractor:`, or
  `generic-explanation:`

Recommended thresholds:

```text
duplicateQuestionRate >= 40
genericDistractors >= 1
genericExplanations >= 1
```

Output behavior:

- Regenerate quiz files
- Preserve document files when possible
- Rebuild metadata and ZIP after quiz regeneration
- Re-run quality report on the merged result

Prompt direction:

- Vary the subtopic, use case, and concept tested by each question
- Generate theme-specific plausible wrong answers
- Explain why the correct answer is correct and why wrong choices fail

### full regeneration

Use when both document and quiz quality are poor, or when the generator cannot
reliably isolate the faulty target.

Primary triggers:

- `duplicateDocumentRate >= 50` and `duplicateQuestionRate >= 40`
- Score is below 40
- The same target-specific retry already failed
- Pack structure makes partial regeneration unsafe

Output behavior:

- Regenerate the whole pack
- Rebuild all output files
- Re-run quality report
- Keep the final generated result even if quality remains low

Prompt direction:

- Re-plan the pack structure before writing files
- Assign each document and quiz a distinct role
- Avoid copying the same stem, choices, explanation, or section outline

## Decision examples

| Finding | Retry target | Reason |
|---|---|---|
| `duplicateQuestionRate` is high | `quiz` | Question stems, explanations, or choices are repeated. |
| `duplicateDocumentRate` is high | `document` | Documents share content or section structure. |
| `genericDistractors` is high | `quiz` | Wrong choices need topic-specific alternatives. |
| `genericExplanations` is high | `quiz` | Explanations need topic-specific reasoning. |
| Low score from quiz-only issues | `quiz` | Smallest effective retry target. |
| Low score from document-only issues | `document` | Smallest effective retry target. |
| Low score from mixed issues | `all` | Partial retry may leave the pack incoherent. |

Recommended target selection order:

1. If both document and quiz issues cross their thresholds, choose `all`.
2. If only document issues cross threshold, choose `document`.
3. If only quiz issues cross threshold, choose `quiz`.
4. If score is below 60 but no single metric crosses threshold, choose the target
   with the largest score penalty.
5. If the largest penalty is tied, choose `quiz` first because quiz issues are
   usually more visible to learners and cheaper to regenerate.

## Retry count

Recommended limits:

| Setting | Value | Meaning |
|---|---:|---|
| Default retry count | `1` | One improvement attempt when retry is enabled. |
| Maximum allowed retry count | `2` | Hard upper bound. |
| Values above max | Reject or clamp to `2` | Prevent runaway loops. |

Rules:

- Never retry indefinitely.
- Never retry more than the configured maximum.
- Never allow a configured maximum above `2`.
- Re-run quality evaluation after every retry.
- Stop early if the result reaches `pass`.
- If retry improves score but still remains below the threshold, do not continue
  past the configured maximum.
- If retry makes the score worse, keep the last generated result but record that
  the retry did not improve quality.

Failure behavior:

- Keep the final generated result.
- Write `quality-report.json`.
- Record `retryFailed` or equivalent retry metadata.
- Do not delete the original output unless the implementation also stores enough
  metadata to explain which result is final.

Recommended final-result policy:

- If retry improves the score, final output should be the retry result.
- If retry does not improve the score, final output may still be the retry
  result, but `quality-report.json` must say `improved: false`.
- A future implementation may preserve the original attempt under an internal
  diagnostics directory, but that should be a separate option to avoid cluttering
  the normal output directory.

## Retry prompt design

The retry prompt should include:

- Original theme/title/description/profile
- Reference-mode constraints
- The failed quality metrics
- The warning summaries
- The selected retry target
- A concise improvement instruction matched to the issue
- A reminder to keep the existing JSON schema and SokQA compatibility

### Duplicate questions

Japanese instruction:

```text
問題文が重複しています。各問題で異なる観点・用語・ケースを扱ってください。
同じ問題文、同じ選択肢セット、同じ解説構造を繰り返さないでください。
```

English instruction:

```text
The quiz questions are too similar. Each question must test a different angle,
term, or use case. Do not repeat the same stem, choice set, or explanation
structure.
```

### Generic distractors

Japanese instruction:

```text
誤答肢が汎用的です。テーマに関連する、もっともらしい誤答を生成してください。
「確認しない」「曖昧にする」のような汎用行動ではなく、学習テーマ内で実際に混同しやすい用語・概念・判断を使ってください。
```

English instruction:

```text
The distractors are too generic. Generate plausible wrong answers that are
specific to the theme. Avoid generic behaviors such as skipping basics or using
unclear wording.
```

### Generic explanations

Japanese instruction:

```text
解説が抽象的です。正解理由と誤答が誤りである理由をテーマ固有の内容で説明してください。
単に「最も合っている」ではなく、どの概念・条件・事例に基づいて正しいのかを書いてください。
```

English instruction:

```text
The explanations are too abstract. Explain why the correct answer is correct and
why the wrong answers are wrong using theme-specific concepts, conditions, or
examples.
```

### Duplicate documents

Japanese instruction:

```text
複数ドキュメントが同じ内容です。doc_01は基礎、doc_02は応用、doc_03は実践例のように役割を分散してください。
同じ見出し構造や同じ本文展開を繰り返さないでください。
```

English instruction:

```text
The documents are too similar. Split the roles clearly, for example doc_01 for
basics, doc_02 for applied knowledge, and doc_03 for practical examples. Do not
reuse the same headings or body structure.
```

### Low score with mixed issues

Japanese instruction:

```text
品質スコアが低いため、パック全体を再設計してください。ドキュメントごとの役割、クイズごとの出題観点、各問題の狙いを先に分けてから生成してください。
重複した本文、重複した問題、汎用的な誤答肢、抽象的な解説を避けてください。
```

English instruction:

```text
The quality score is low. Redesign the pack before generating. Separate document
roles, quiz angles, and question intents first. Avoid duplicate content, repeated
questions, generic distractors, and abstract explanations.
```

## quality-report extension proposal

Future `quality-report.json` should keep existing fields for compatibility and
add machine-readable status and retry metadata.

Proposed structure:

```json
{
  "score": 72,
  "status": "warning",
  "duplicateQuestionRate": 35,
  "duplicateDocumentRate": 0,
  "genericDistractors": 2,
  "genericExplanations": 1,
  "retry": {
    "attempted": true,
    "count": 1,
    "max": 1,
    "target": "quiz",
    "reason": "genericDistractors",
    "initialScore": 54,
    "finalScore": 72,
    "improved": true,
    "retryFailed": false
  },
  "warnings": []
}
```

Recommended field meanings:

| Field | Type | Meaning |
|---|---|---|
| `status` | `pass \| warning \| retry_candidate \| failed` | Quality status derived from score and retry outcome. |
| `retry.attempted` | boolean | Whether automatic retry ran. |
| `retry.count` | number | Number of retry attempts actually used. |
| `retry.max` | number | Configured retry maximum after validation/clamping. |
| `retry.target` | `document \| quiz \| all \| null` | Regeneration target selected for the latest retry. |
| `retry.reason` | string | Primary metric or warning category that triggered retry. |
| `retry.initialScore` | number | Score before retry. |
| `retry.finalScore` | number | Score after retry. |
| `retry.improved` | boolean | Whether final score is higher than initial score. |
| `retry.retryFailed` | boolean | Whether retry was attempted but quality still failed the configured threshold. |

Compatibility rule:

- Existing top-level fields must remain stable.
- New consumers should read `status` and `retry`.
- Old consumers can continue reading `score` and `warnings`.

Optional future diagnostics:

```json
{
  "retryHistory": [
    {
      "attempt": 1,
      "target": "quiz",
      "reason": "duplicateQuestionRate",
      "scoreBefore": 48,
      "scoreAfter": 70
    }
  ]
}
```

`retryHistory` is useful, but not required for the first implementation.

## CLI option proposal

Recommended options:

| Option | Example | Meaning |
|---|---|---|
| `--quality-retry` | `--quality-retry` | Enable automatic retry for low-quality output. Default is off. |
| `--quality-retry-max` | `--quality-retry-max 2` | Maximum retry attempts. Valid range should be `0..2`. |
| `--quality-threshold` | `--quality-threshold 80` | Minimum score considered acceptable. Default should be `80`. |
| `--fail-on-low-quality` | `--fail-on-low-quality` | Exit non-zero if final quality remains below threshold. |

Recommended behavior:

- Without `--quality-retry`, the CLI should generate once and write the report.
- With `--quality-retry`, the CLI should retry only when the score is below the
  retry trigger threshold.
- `--quality-threshold` should control final pass/fail status.
- `--fail-on-low-quality` should affect exit code, not whether files are written.
- `--quality-retry-max 0` should be valid and should mean "evaluate but do not
  retry".
- `--quality-retry-max` values above `2` should be rejected with a clear message
  or clamped to `2`. Rejection is clearer for CLI users.

Recommended defaults:

```text
qualityRetry: false
qualityRetryMax: 1
qualityThreshold: 80
failOnLowQuality: false
```

Potential future config fields:

```json
{
  "quality": {
    "retry": true,
    "retryMax": 1,
    "threshold": 80,
    "failOnLowQuality": false
  }
}
```

If both config and CLI are present, CLI should override config.

## Implementation notes

### Keep partial regeneration explicit

Partial regeneration should not silently mix incompatible content. If quiz-only
regeneration depends on document content, the retry prompt should include the
current documents as context.

### Preserve reference-mode constraints

`exact_text_document` is a special case:

- It should not generate quiz files.
- It should not rewrite source text.
- Automatic regeneration should probably be disabled or limited to reporting
  document duplication, because exact text conversion may intentionally preserve
  source similarity.

`source_only` should remain source-constrained:

- Retry prompts must not ask the model to add external facts.
- Improvements should focus on organization, separation, and faithful question
  variation based on the source.

### Avoid warning-string coupling

Warning prefixes are useful, but retry logic should primarily use structured
metrics:

- `duplicateQuestionRate`
- `duplicateDocumentRate`
- `genericDistractors`
- `genericExplanations`
- Future structured issue codes

Warnings may be included as prompt context, but should not be the only retry
decision input.

### Make retry deterministic enough to debug

The report should record:

- Whether retry ran
- Why it ran
- What target was regenerated
- How many attempts were used
- Whether the score improved

Without these fields, users may not understand why generation took longer or why
the final result still has warnings.

### Do not hide failed quality

Retry should improve output when possible, but it must not mask final quality
problems. If final output remains below threshold, the quality report should make
that clear and `--fail-on-low-quality` should allow automation to fail the run.

### Keep cost and latency visible

Automatic retry can increase LLM calls. Therefore:

- Default should remain no retry unless explicitly enabled.
- Maximum retry count should be small.
- CLI output should mention retry attempts when implemented.

## v0.8.3 and later candidates

Recommended follow-up sequence:

1. Add `status` to `quality-report.json` without changing generation behavior.
2. Add retry metadata fields with `attempted: false` by default.
3. Add CLI parsing for retry-related options.
4. Implement quiz-only retry.
5. Implement document-only retry.
6. Implement full retry for mixed failures.
7. Add tests for target selection and retry limit enforcement.
8. Add config-file support for quality retry settings.
9. Add optional retry history for diagnostics.

Possible future improvements:

- Per-warning structured issue codes
- Configurable metric weights
- Separate thresholds for document and quiz quality
- Cost budget controls for LLM-backed generation
- Optional preservation of original failed attempts for debugging
- Quality-specific prompt templates by language

## Recommended v0.8.2 decision summary

- Pass line: `score >= 80`
- Warning range: `60..79`
- Retry candidate: `< 60`
- Default retry count when enabled: `1`
- Absolute max retry count: `2`
- Retry target selection: smallest affected target first
- Quiz retry triggers: duplicate questions, generic distractors, generic
  explanations
- Document retry trigger: duplicate documents
- Full retry trigger: mixed severe issues or very low score
- Failed retry behavior: keep final output and record retry failure
- CLI retry default: off
