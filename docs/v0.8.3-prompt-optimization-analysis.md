# v0.8.3 Prompt Optimization Analysis

This document analyzes the current generation "prompt" structure and designs a
future prompt optimization strategy for Studio-level output quality.

Scope for v0.8.3:

- Analysis and design only
- No implementation
- No code changes
- No README changes
- No test changes

---

## Summary

The current Factory does not contain an explicit LLM prompt layer. There are no
separate system prompts, user prompts, or validation prompts. Instead, output is
produced by deterministic TypeScript templates in `lib/learning-pack.ts`.

That means the current quality gap is not only a "weak prompt" problem. It is a
missing prompt-orchestration problem:

- Document generation is scaffold-based, not content-generation-based.
- Quiz generation repeats the same stem, answer pattern, distractor pool, and
  explanation template.
- Metadata generation copies config values and file lists, but does not create
  topic-specific learning metadata.
- TTS generation wraps raw text in a single language prefix and punctuation
  suffix, but does not handle pronunciation, inline language switching, or
  abbreviations.
- Validation is schema/consistency validation only, not LLM-based content review.

The highest-impact next step is to introduce a structured generation prompt that
asks for a content plan first, then generates documents, quizzes, metadata, and
TTS under explicit quality constraints.

## Current prompt structure

### Explicit prompts

Current state:

| Prompt type | Exists? | Notes |
|---|---:|---|
| System prompt | No | No LLM client or system message exists in the current generator. |
| User prompt | No | Config values are passed to deterministic builders, not to an LLM. |
| Document prompt | No | Document output comes from fixed templates and reference splitting. |
| Quiz prompt | No | Quiz output comes from fixed templates and deterministic pools. |
| Metadata prompt | No | Metadata is assembled from config and generated file names. |
| TTS prompt | No | TTS is generated by string normalization and prefix/suffix helpers. |
| Validation prompt | No | Validation is programmatic shape and consistency checking. |

For v0.8.3 analysis, "current prompt" means the current fixed template behavior
that would need to become prompt instructions in a future LLM-backed generator.

### Document generation

Current implementation areas:

- `buildGenericDocumentItems`
- `buildSourceOnlyDocumentItems`
- `buildSourcePlusDocumentItems`
- `buildExactTextDocumentItems`
- `buildDocumentFile`

Current generation by reference mode:

| Mode | Current behavior |
|---|---|
| `none` | Emits fixed scaffold items: intro, theme, goal, key points, optional reading/detail/example/difficulty sections, summary, practice tip. |
| `source_only` | Splits reference text into non-empty lines and outputs those lines exactly. |
| `source_plus` | Emits first 6 reference lines, then adds a generated summary and practice tip. |
| `exact_text_document` | Splits reference into paragraphs and emits them exactly. Quiz generation is suppressed. |

Current fixed templates include:

```text
Intro (${toneLabel}): ${intro}
Theme: ${theme}
Goal: ${title}
Key Points: ${description || theme}
Reading note (${vocabularyRegister} vocabulary): ...
Detail/Deep Dive: This pack is designed for ${audienceLabel} learners.
Example N: ...
Challenge: Apply ${theme} to a non-obvious, real-world case.
Basics: Start from the simplest point of ${theme}.
Summary: ...
Study/Training/Exam Tip: ...
```

Profile reflection:

| Profile field | Current document effect |
|---|---|
| `targetUser` | Changes `audienceLabel` and `vocabularyRegister` wording. |
| `difficulty` | Adds `Basics` for easy or `Challenge` for hard. |
| `learningStyle` | Adds reading note for `reading`; affects short-sentence path for `audio`; `quiz` adds no document structure beyond quiz stem cue. |
| `outputMode` | Changes practice tip: study/training/exam. |
| `tone` | Changes intro, summary, and tone label. |
| `detailLevel` | Controls `Key Point` / `Detail` / `Deep Dive` label and whether detail block appears. |
| `exampleLevel` | Controls number of examples. |
| `audioOptimization` | Shortens TTS/document strings through `shortenForAudio`. |
| `practicalExamples` | Switches placeholder examples to three concrete-ish fixed examples. |
| `sentenceLength` | `short` triggers the short-sentence path. |

Language reflection:

- `language` is copied to JSON.
- TTS prefix is `[ja-JP]` when language starts with `ja`; otherwise `[en-US]`.
- Document body language is not actively generated from `language`; current
  templates are mostly English even when `language` is `ja`.

### Quiz generation

Current implementation areas:

- `buildGenericQuizQuestions`
- `buildSourceOnlyQuizQuestions`
- `quizQuestionTemplate`
- `choiceA`
- `themeDistractors`
- `referenceDistractors`
- `explanationTemplate`
- `buildQuizFile`

Current fixed templates include:

```text
Question N of TOTAL: Select the best option regarding ${theme}.
Case study: Which action best handles ${theme}?
How should you apply ${theme} in practice?
Which choice best aligns with the theme: ${theme}?
```

Correct answer templates:

```text
Focus on the core concept of ${theme} ...
Apply ${theme} in a realistic situation ...
Decide the best action for ${theme} ...
```

Distractor templates:

```text
Ignore customer context
Skip confirmation steps
Use unclear language
Skip the basics of ${theme}
Apply ${theme} without checking context
Use unclear wording when explaining ${theme}
```

Explanation templates:

```text
Why/Reason/Rationale: Best matches ${theme}.
Why/Reason/Rationale: It directly matches the theme: ${theme}, fitting the ${quizStyle} question.
Why/Reason/Rationale: The best option applies ${theme} correctly. The other options fail because ...
```

Profile reflection:

| Profile field | Current quiz effect |
|---|---|
| `targetUser` | Changes explanation prefix only. |
| `difficulty` | Adds basics/applied cue and changes correct-answer suffix. |
| `learningStyle` | Adds `"Review carefully."` or `"Use the reading material."` to stems. |
| `outputMode` | `exam` uses numbered `"Question N of TOTAL"` stem. |
| `quizStyle` | Chooses concept/application/case-study stem and correct-answer style. |
| `distractorSource` | Chooses fixed, theme, or reference-line distractor pool. |
| `explanationDepth` | Chooses short/standard/detailed explanation template. |

Language reflection:

- Quiz body strings are not translated by `language`.
- TTS prefix follows `language`.

Known structural issues:

- `answerIndex` is always `0`.
- The same stem shape is reused for every question.
- Distractor pools repeat cyclically.
- Explanations are template-level and repeat across questions.
- Multiple quiz files on the same config repeat the same question set.

### Metadata generation

Current implementation areas:

- `buildMetadataFile`
- `buildDocumentFileName`
- `buildQuizFileName`
- `serializeLearningPackFiles`

Current behavior:

- Copies `id`, `title`, `description`, `language`, source info, references, and
  resolved `profile`.
- Sets `generator` to `sokqa-learning-pack-factory`.
- Sets `version` to `0.7.0`.
- Builds document and quiz file lists from generated counts.

There is no LLM-generated metadata prompt. Metadata is structurally reliable but
not content-rich.

Missing Studio-like metadata behaviors:

- Topic-specific `globalTags`
- Chapter-specific document titles
- Learning-objective descriptions
- Optional author
- Optional asset base URL

### TTS generation

Current implementation areas:

- `ttsPrefix`
- `toTts`
- `ensureEndsWithComma`
- `shortenForAudio`
- `buildQuizTts`

Current behavior:

- Prefixes all TTS text with `[ja-JP]` for Japanese or `[en-US]` otherwise.
- Adds a Japanese comma for Japanese and an English comma for other languages.
- Then `ensureEndsWithComma` may append an English comma again.
- Quiz TTS is always generated for question, choices, answer, and explanation.
- Choice TTS intentionally omits choice numbers.

Current TTS has no prompt-level handling for:

- Pronunciation
- Inline language switching
- Abbreviation reading
- Kana reading for difficult words
- Selective omission of TTS for visual-only items
- Natural sentence-final punctuation by language

### Validation

Current implementation areas:

- `validateLearningPack`
- `validateDocument`
- `validateQuiz`
- `validateMetadata`
- `validateGenerationCounts`
- `validateSerializedFiles`
- `quality-report.json` generation in `lib/learning-pack-zip.ts`
- quality checks in `lib/quality-check.ts`

Current validation is programmatic. It checks:

- JSON serializability
- Required document fields
- Required quiz fields
- Four choices per question
- `answerIndex` range
- No choice numbering in `tts.choiceTexts`
- Metadata consistency
- File list consistency
- Generation counts
- Quality metrics after export

There is no validation prompt that asks an LLM to inspect content quality,
pedagogy, factuality, or learner suitability.

## Prompt structure analysis

### Document

| Aspect | Current state | Problem |
|---|---|---|
| Learning objective | Mostly copied from config title/description. | Does not create a fresh learner-facing objective per document. |
| Output format | Valid JSON shape is deterministic. | Shape is reliable, but content inside items is thin. |
| Writing style | Tone changes intro/summary labels. | Tone does not affect substance, vocabulary, or explanation depth enough. |
| Length | No content-length target beyond fixed item count. | Documents are much shorter than Studio samples. |
| Section structure | Fixed scaffold. | Multiple documents repeat the same structure. |
| Examples | Fixed customer-service-like examples or placeholder examples. | Examples are not reliably domain-specific. |
| Learning sequence | Intro/theme/goal/key points/detail/example/summary/tip. | No topic-specific learning arc. |

Future prompt requirement:

- Ask the model to design a document-level learning arc before writing.
- Require each document to have a unique role.
- Require each item to teach one concrete concept, fact, rule, example, or
  scenario.
- Require enough density to resemble Studio quality without becoming verbose.

### Quiz

| Aspect | Current state | Problem |
|---|---|---|
| Question generation | One stem template repeated with index changes. | Low variation and high duplicate rate. |
| Choice generation | One correct template plus three cyclic distractors. | Choices are obvious and not domain-specific. |
| Explanation generation | One template selected by depth. | Explanations restate the theme rather than teaching. |
| Distractor generation | Fixed/theme/reference line pools. | Theme distractors are generic bad behaviors, not plausible wrong facts. |
| `answerIndex` | Always `0`. | Quiz can be gamed without reading. |
| Difficulty | Adds labels such as basics/applied. | Does not require deeper reasoning or more precise knowledge. |

Future prompt requirement:

- Ask for a quiz blueprint that assigns a different subtopic, skill, or scenario
  to every question.
- Require plausible distractors from the same domain category as the correct
  answer.
- Require explanation to mention the correct rule/fact and at least one reason
  the distractors fail.
- Require answer positions to be distributed across `0..3`.

### TTS

| Aspect | Current state | Problem |
|---|---|---|
| Pronunciation | Raw text passthrough. | Abbreviations and difficult terms may be read poorly. |
| Language | One prefix for the whole string. | Mixed-language learning content is misread. |
| Inline language | Not supported. | English phrases in Japanese packs lack `[en-US]` segments. |
| Abbreviation | No expansion map or prompt instruction. | Terms like `IT`, `URL`, `SSL/TLS` are not spoken naturally. |
| Selectivity | TTS emitted for almost every item. | Visual-only or label-like items get noisy narration. |
| Punctuation | Mechanical suffixing. | Japanese can produce unnatural punctuation. |

Future prompt requirement:

- Separate display text from spoken text.
- Ask for inline language tags only where needed.
- Require expansion of abbreviations and difficult readings in `tts` fields.
- Allow TTS omission or short narration for visual-only/structural items if the
  schema permits it in a future version.

## Studio gap analysis

This section builds on `docs/v0.8-studio-gap-analysis.md` and frames the gap as
prompt-design requirements.

### Document gap

| Dimension | Studio | Factory | Prompt implication |
|---|---|---|---|
| Information amount | Dense, concept-rich paragraphs. | Short scaffold lines. | Prompt must specify minimum concept density and item count. |
| Structure | Topic-specific learning arc. | Fixed scaffold. | Prompt must ask for a content plan and document roles. |
| Depth | Explains concepts, risks, examples, and defenses. | Mentions the theme. | Prompt must require concrete concepts and supporting details. |
| Examples | Domain-specific and memorable. | Generic or customer-service-flavored. | Prompt must require examples from the theme/reference. |
| Learning order | Concepts build progressively. | Same order for every topic. | Prompt must require beginner-to-advanced sequencing. |

### Quiz gap

| Dimension | Studio | Factory | Prompt implication |
|---|---|---|---|
| Variation | Questions cover many facts and cases. | Same stem repeats. | Prompt must require unique question intents. |
| Difficulty | Questions require topic knowledge. | Difficulty labels only. | Prompt must define easy/normal/hard behavior. |
| Distractors | Plausible same-domain alternatives. | Generic wrong behaviors. | Prompt must constrain distractors by category and plausibility. |
| Explanations | Teach the fact behind the answer. | Restate the theme. | Prompt must require answer-specific explanation. |
| Duplicate rate | Low. | High. | Prompt must prohibit repeated stems, choice sets, and explanations. |
| Answer positions | Distributed. | Always index 0. | Prompt/output contract must require answer distribution. |

### TTS gap

| Dimension | Studio | Factory | Prompt implication |
|---|---|---|---|
| Reading quality | Natural, language-aware. | Raw prefix/suffix. | Prompt must generate spoken text intentionally. |
| Abbreviations | Expanded in Japanese readings. | Raw abbreviations remain. | Prompt or post-process must expand key terms. |
| Multilingual | Inline `[en-US]` / `[ja-JP]`. | Single prefix per field. | Prompt must support inline code-switching. |
| Pronunciation | Handles difficult readings. | No reading hints. | Prompt must ask for reading-friendly TTS. |
| Selectivity | TTS appears where useful. | TTS everywhere. | Future schema/prompt should distinguish display-only vs narration. |

## Problems

### P1: No content planning step

The generator writes final output directly. Without a plan, it cannot allocate
different document roles, quiz subtopics, or question intents.

Impact:

- Duplicate documents
- Duplicate quizzes
- No learning progression
- Profile settings only change wording

### P2: Profile fields affect labels more than substance

The profile system is structurally useful, but most fields currently map to
labels, prefixes, or small sentence variations.

Impact:

- `difficulty: hard` does not create hard questions.
- `targetUser: beginner` does not truly simplify explanations.
- `quizStyle: case-study` changes the stem but not the underlying case quality.

### P3: Distractors are not generated as domain alternatives

The current theme distractors describe generic poor behavior rather than wrong
answers from the same topic.

Impact:

- Learners can guess by tone.
- Questions do not test knowledge.
- `genericDistractors` quality warnings increase.

### P4: Explanations do not teach

Short and standard explanations usually restate that the answer matches the
theme.

Impact:

- Learners do not understand why the answer is correct.
- Wrong choices are not corrected.
- `genericExplanations` warnings increase.

### P5: TTS is formatting, not pronunciation design

Current TTS is derived mechanically from display text.

Impact:

- Japanese TTS can read English terms incorrectly.
- Language-learning packs lose their core value.
- Audio-optimized mode may produce label-only narration.

## Improvement candidates

### Priority A

These have the largest quality impact and should be considered first.

| Candidate | Reason |
|---|---|
| Add a content planning prompt | Prevents duplicated documents/quizzes by assigning roles and subtopics before generation. |
| Replace scaffold document prompt with learning-arc prompt | Moves output from theme-string slots to actual educational content. |
| Replace quiz template prompt with question-blueprint prompt | Ensures each question tests a distinct concept, use case, or fact. |
| Generate domain-specific distractors | Directly improves quiz usefulness and reduces generic distractor warnings. |
| Generate answer-specific explanations | Converts explanations from labels into teaching moments. |
| Require answer index distribution | Removes the always-0 guessing pattern. |
| Add TTS pronunciation instructions for mixed-language and abbreviations | Critical for Japanese and language-learning packs. |

### Priority B

Useful improvements, but secondary to core content quality.

| Candidate | Reason |
|---|---|
| Add prompt-level JSON self-check section | Helps the model verify counts, schema, and uniqueness before returning. |
| Generate topic-specific tags | Improves discovery and Studio parity. |
| Generate chapter-style document/quiz titles | Helps learners understand where each file fits. |
| Add language-specific style guides | Improves naturalness after the core prompt structure is stable. |
| Add reference-mode prompt variants | Keeps `source_only`, `source_plus`, and `exact_text_document` constraints explicit. |

### Priority C

Future candidates after core prompt quality works.

| Candidate | Reason |
|---|---|
| Multi-pass generation with critique | Higher quality, but increases latency and cost. |
| Per-profile prompt packs | Useful once base prompts are stable. |
| TTS dictionary generation | Better pronunciation control, but may require schema or runtime support. |
| Prompt A/B evaluation fixtures | Good for measuring prompt versions over time. |
| Model-specific prompt variants | Useful only after a provider/model strategy exists. |

## Prompt improvement proposals

### Document

Current:

```text
Theme: ${theme}
Goal: ${title}
Key Points: ${description || theme}
Detail: This pack is designed for learners.
Example: Apply ${theme} to a practical scenario.
```

Improvement proposal:

```text
You are generating SokQA document content for learners.

Create a short content plan first:
- Assign each document a unique role.
- Assign each document 6-10 learning items.
- Each item must teach one concrete fact, concept, process, risk, example, or
  action related to the theme.
- The items must progress from basic understanding to practical use.
- Do not repeat the same section structure across documents.

Then write document JSON items using the required schema.
Use the configured language.
Use the configured profile:
- targetUser controls vocabulary and assumed prior knowledge.
- difficulty controls conceptual depth and example complexity.
- learningStyle controls whether items are optimized for reading, audio, quiz
  preparation, or balanced study.
- tone controls wording, not content quality.
- practicalExamples requires real theme-specific examples.
```

Expected effect:

- More topic-specific content
- Clearer learning sequence
- Lower duplicate document rate
- Better profile fidelity

### Quiz

Current:

```text
Question N: Select the best option regarding ${theme}.
Correct: Focus on the core concept of ${theme}.
Wrong: Skip the basics of ${theme}.
Wrong: Apply ${theme} without checking context.
Wrong: Use unclear wording when explaining ${theme}.
Explanation: Best matches ${theme}.
```

Improvement proposal:

```text
Create a quiz blueprint before writing questions.

For each question:
- Assign a unique tested concept, fact, scenario, or misconception.
- Do not repeat the same stem pattern.
- Generate one correct answer and three plausible wrong answers.
- Wrong answers must be from the same topic category as the correct answer.
- Avoid generic behavior distractors unless the topic is explicitly behavioral.
- Distribute answerIndex across 0, 1, 2, and 3.
- Explanation must state why the correct answer is correct and why at least one
  distractor is wrong.

Apply quizStyle:
- concept-check: test definitions, relationships, and key facts.
- application: test what the learner should do in a realistic situation.
- case-study: present a short case and ask for the best judgment.

Apply difficulty:
- easy: direct recognition of basic facts.
- normal: one-step application or distinction.
- hard: scenario-based reasoning or common misconception traps.
```

Expected effect:

- Lower duplicate question rate
- Better distractor plausibility
- Better explanation quality
- More meaningful difficulty

### Metadata

Current:

```text
title = config.title
description = config.description || "Generated learning pack"
globalTags = ["sokqa-learning-pack-factory"]
```

Improvement proposal:

```text
Generate learner-facing metadata from the content plan:
- Document titles should reflect chapter or role.
- Quiz titles should reflect checked skill or chapter.
- Description should state what the learner can do after completing the pack.
- globalTags should include 3-5 topic-specific tags.
- Keep schema compatibility and existing required fields.
```

Expected effect:

- Better Studio parity
- Better discoverability
- More useful file titles in multi-document packs

### TTS

Current:

```text
tts = prefix(language) + displayText + punctuation
```

Improvement proposal:

```text
Generate TTS as spoken text, not as a copy of display text.

Rules:
- Use the configured language as the default voice.
- Use inline language tags for foreign phrases, such as [en-US]Good morning[ja-JP].
- Expand abbreviations in spoken form when the target language requires it.
- Add pronunciation-friendly readings for difficult terms.
- Keep choiceTexts free of numbering.
- Avoid label-only narration such as "Theme" or "Example 1".
- Use natural punctuation for the target language.
```

Expected effect:

- Better Japanese TTS
- Better language-learning packs
- Fewer unnatural audio strings

### Validation

Current:

```text
Programmatic schema and consistency checks.
Quality report after output.
No LLM validation prompt.
```

Improvement proposal:

```text
Before finalizing, self-check the generated JSON:
- All counts match config.
- Every document has a distinct role.
- No repeated document item structure.
- Every quiz question has a unique intent.
- No repeated choice sets.
- answerIndex values are distributed.
- Explanations mention theme-specific facts.
- TTS uses language tags and pronunciation rules where needed.
```

Expected effect:

- Fewer obvious generation defects before programmatic validation
- Better alignment with v0.8 quality-report metrics

## Proposed future prompt architecture

### Phase 1: Content plan

Purpose:

- Allocate document roles
- Allocate quiz coverage
- Decide key concepts, examples, misconceptions, and TTS needs

Output:

```json
{
  "documentPlan": [
    {
      "id": "doc_01",
      "role": "basics",
      "learningGoal": "...",
      "items": ["...", "..."]
    }
  ],
  "quizPlan": [
    {
      "id": "q-1",
      "testedConcept": "...",
      "difficulty": "easy",
      "distractorStrategy": "same-category plausible alternatives"
    }
  ],
  "ttsPlan": {
    "defaultLanguage": "ja-JP",
    "inlineLanguages": ["en-US"],
    "pronunciationTerms": ["IT", "URL"]
  }
}
```

### Phase 2: Structured generation

Purpose:

- Generate final document, quiz, metadata, and TTS JSON from the plan.

Rules:

- Use the exact schema.
- Preserve source constraints.
- Match counts.
- Use profile as content requirements.

### Phase 3: Self-check

Purpose:

- Catch obvious quality and schema defects before programmatic validation.

Checks:

- Count checks
- Duplicate-intent checks
- Distractor plausibility checks
- Explanation specificity checks
- TTS language/pronunciation checks

### Phase 4: Programmatic validation and quality report

Purpose:

- Keep current deterministic validation and quality-report checks as the final
  gate.

This keeps LLM judgment useful but not trusted as the only validator.

## Implementation candidates after v0.8.3

Recommended order:

1. Add prompt design fixtures under docs or examples without runtime use.
2. Add a content-plan schema for future LLM output.
3. Add an optional LLM generation path behind an explicit flag.
4. Implement document prompt first for `mode: none`.
5. Implement quiz prompt with answer-index distribution and unique question
   intents.
6. Add TTS prompt rules for Japanese and mixed-language content.
7. Add prompt-level self-check output.
8. Connect the v0.8.2 automatic regeneration design to prompt-specific retry
   instructions.
9. Add evaluation fixtures comparing Factory output to Studio-like expectations.

Do not replace current deterministic validation. The LLM path should feed into
the existing schema validation, ZIP export, and quality-report pipeline.

## Recommended v0.8.3 decision summary

- Current prompts: none; current generation is deterministic templates.
- Main gap: missing content planning and LLM-quality generation instructions.
- Priority A: document learning arc, quiz blueprint, domain distractors,
  answer-specific explanations, answer-index distribution, TTS pronunciation.
- Priority B: metadata richness, JSON self-check, reference-mode prompt variants.
- Priority C: multi-pass critique, prompt A/B evaluation, model-specific prompts.
- Profile status: all fields are represented, but mostly as wording flags.
- Future strategy: plan first, generate second, self-check third, validate
  programmatically last.
